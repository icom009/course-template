이 차시의 핵심 용어

Spring LegacyXML 설정 파일을 직접 작성하는 전통적 Spring 사용 방식. 본 과정의 기반.
SpringBootSpring 위에 자동 설정과 내장 톰캣을 얹은 도구. 「설정만 미리 깔린 Spring」.
@SpringBootApplication자동 설정·컴포넌트 스캔·Configuration 세 가지를 묶은 어노테이션. 한 줄로 시작점이 됨.
Auto-Configuration클래스패스에 어떤 라이브러리가 있는지를 보고, 필요한 빈을 SpringBoot 가 알아서 등록하는 메커니즘.
Starter함께 잘 동작하는 의존성을 미리 묶어둔 세트. spring-boot-starter-web 한 줄이 webmvc + jackson + tomcat-embed 를 모두 가져옴.
임베디드 톰캣 (Embedded Tomcat)jar 안에 톰캣이 들어 있어서 외부 톰캣 설치·war 배포 없이 java -jar 한 줄로 실행되는 구조.
application.propertieskey=value 형식의 단일 설정 파일. Legacy 의 7개 XML 자리를 한 파일이 대신함.

1. 이전 과정의 끝에서 — 왜 이게 필요한가

v∞ 까지 가본 사람만 아는 피로

Hello Servlet (v0) 부터 REST API 게시판 (v∞) 까지 — 우리는 끝까지 갔습니다. 그런데 새 프로젝트를 시작하려고 하면 그 모든 설정을 처음부터 다시 써야 합니다.

web.xml, servlet-context.xml, root-context.xml, pom.xml, ViewResolver, 인코딩 필터 ··· 코드 한 줄 짜기 전에 30분이 사라집니다.

본 과정에서는 그 부담을 일부러 짊어졌습니다 — 「되긴 하는데 왜 되는지 모르는」 상태를 피하기 위해서. 하지만 현장에서는 매번 그렇게 시작하지 않습니다. 오늘은 현장이 어떻게 그 부담을 던졌는지를 봅니다.

2. Legacy 가 들고 있던 7개의 XML

본 과정에서 직접 만져본 — 또는 만질 뻔했던 — 설정 파일들입니다. 새 프로젝트를 만들 때마다 이 7개가 따라옵니다.

Legacy 의 설정 7종 세트 — 한 줄 요약
  • web.xml — 톰캣이 가장 먼저 읽는 파일. DispatcherServlet 등록, 인코딩 필터 등록, 시작 URL 패턴 매핑.
  • servlet-context.xml — 웹 계층(컨트롤러·뷰리졸버·정적 리소스) 빈 설정. /WEB-INF/views/ 같은 JSP 위치도 여기에.
  • root-context.xml — 백엔드 계층(DataSource·SqlSessionFactory·Service·DAO) 빈 설정. DB 연결 정보가 여기에 들어감.
  • pom.xml — Maven 의존성과 자바 버전, 빌드 플러그인 명시. 버전 번호 한 자릿수 어긋나면 빌드 실패.
  • mybatis-config.xml — MyBatis 의 typeAlias·setting (camelCase 매핑 등). 본 과정에서 한 번 만진 그 파일.
  • log4j.xml / logback.xml — 콘솔과 파일에 어떤 로그를 어떤 형식으로 찍을지. SQL 로그 켜는 자리.
  • (서버) Tomcat 의 server.xml / context.xml — 포트, 인코딩, 컨텍스트 패스. WAS 자체의 설정.

2-1. web.xml — 톰캣의 첫 페이지

톰캣이 부팅하면서 가장 먼저 읽는 파일입니다. 「누가 요청을 받을 것인가」 를 여기에서 결정합니다.

<!-- web.xml 발췌 -->
<servlet>
  <servlet-name>appServlet</servlet-name>
  <servlet-class>org.springframework.web.servlet.DispatcherServlet</servlet-class>
  <init-param>
    <param-name>contextConfigLocation</param-name>
    <param-value>/WEB-INF/spring/appServlet/servlet-context.xml</param-value>
  </init-param>
</servlet>
<servlet-mapping>
  <servlet-name>appServlet</servlet-name>
  <url-pattern>/</url-pattern>
</servlet-mapping>

<filter>
  <filter-name>encodingFilter</filter-name>
  <filter-class>org.springframework.web.filter.CharacterEncodingFilter</filter-class>
  <init-param>
    <param-name>encoding</param-name><param-value>UTF-8</param-value>
  </init-param>
</filter>

한 자만 어긋나도 톰캣이 안 뜨거나 한글이 깨집니다. SpringBoot 에서는 — spring-boot-starter-web 한 줄이 들어가면 이 파일 자체가 사라집니다. DispatcherServlet 자동 등록 + UTF-8 기본값이 한 세트로 들어옵니다.

2-2. servlet-context.xml — 웹 계층 빈 설정

본 과정에서 ViewResolver 의 prefix/suffix 를 직접 채워본 그 파일입니다.

<!-- servlet-context.xml 발췌 -->
<beans:bean class="org.springframework.web.servlet.view.InternalResourceViewResolver">
  <beans:property name="prefix" value="/WEB-INF/views/" />
  <beans:property name="suffix" value=".jsp" />
</beans:bean>

<resources mapping="/resources/**" location="/resources/" />
<context:component-scan base-package="com.example.controller" />

SpringBoot 의 application.properties 에서는 다음 두 줄이면 같은 일을 합니다:

spring.mvc.view.prefix=/WEB-INF/views/
spring.mvc.view.suffix=.jsp

component-scan 은 @SpringBootApplication 에 이미 포함되어 있어 따로 적지 않습니다. 정적 리소스 매핑(/resources/**)도 기본값으로 켜져 있습니다(src/main/resources/static/).

2-3. root-context.xml — DB 와 백엔드 빈

본 과정의 데이터·MyBatis Part 에서 가장 길게 손댄 파일입니다.

<!-- root-context.xml 발췌 -->
<bean id="dataSource" class="com.zaxxer.hikari.HikariDataSource">
  <property name="jdbcUrl" value="jdbc:mysql://localhost:3306/myapp" />
  <property name="username" value="root" />
  <property name="password" value="secret" />
  <property name="maximumPoolSize" value="10" />
</bean>

<bean id="sqlSessionFactory" class="org.mybatis.spring.SqlSessionFactoryBean">
  <property name="dataSource" ref="dataSource" />
  <property name="mapperLocations" value="classpath:mappers/*.xml" />
</bean>

<mybatis-spring:scan base-package="com.example.mapper" />
<tx:annotation-driven />

SpringBoot 의 같은 일:

spring.datasource.url=jdbc:mysql://localhost:3306/myapp
spring.datasource.username=root
spring.datasource.password=secret
spring.datasource.hikari.maximum-pool-size=10
mybatis.mapper-locations=classpath:mappers/*.xml

HikariCP 는 starter-jdbc 의 기본 커넥션 풀이라 별도 클래스 지정도 필요 없습니다. @MapperScan 만 Application 클래스에 한 줄 더하면 됩니다.

2-4. pom.xml — 버전 충돌의 그 파일

<!-- Legacy pom.xml — 버전을 다 적던 시절 -->
<dependency>
  <groupId>org.springframework</groupId>
  <artifactId>spring-webmvc</artifactId>
  <version>5.3.20</version>
</dependency>
<dependency>
  <groupId>org.springframework</groupId>
  <artifactId>spring-jdbc</artifactId>
  <version>5.2.8</version>  <!-- 한 줄 어긋남 -->
</dependency>
<dependency>
  <groupId>org.mybatis</groupId>
  <artifactId>mybatis-spring</artifactId>
  <version>2.0.6</version>
</dependency>

SpringBoot 의 같은 자리:

<parent>
  <groupId>org.springframework.boot</groupId>
  <artifactId>spring-boot-starter-parent</artifactId>
  <version>3.2.0</version>
</parent>

<dependency>
  <groupId>org.springframework.boot</groupId>
  <artifactId>spring-boot-starter-web</artifactId>
</dependency>

버전 번호가 한 곳에만 — parent 의 3.2.0. 모든 하위 의존성의 호환 버전이 이 한 숫자에서 결정됩니다. 「mybatis-spring 2.0.6 으로 올렸더니 spring 이 안 맞네…」 같은 야근이 사라집니다.

2-5 ~ 2-7. 나머지 파일들

  • mybatis-config.xml — typeAlias 와 camelCase 매핑 설정 자리. SpringBoot 에서는 mybatis.type-aliases-package / mybatis.configuration.map-underscore-to-camel-case=true 두 줄로 끝.
  • log4j.xml / logback.xml — SpringBoot 는 logback 을 기본으로 켭니다. SQL 로그 한 줄 켜는 것도 logging.level.com.example.mapper=DEBUG 한 줄.
  • Tomcat server.xml — 임베디드 톰캣이 jar 안에 있으므로 외부 server.xml 자체가 없습니다. 포트/컨텍스트는 application.properties 의 server.port, server.servlet.context-path 한 줄씩.

이 7개 중 한 줄만 어긋나면 — 톰캣이 안 뜨거나, 한글이 깨지거나, 404 가 떠도 원인을 짐작하기 힘듭니다. 본 과정에서 이 부담을 떠안고 갔던 이유는 「데이터 흐름이 어디서 끊겼는지」 를 직접 짚을 수 있게 하기 위해서였습니다.

[새 프로젝트 시작 — Legacy] ① 톰캣 설치 / 버전 맞추기 ② Eclipse 에서 Server 등록 ③ pom.xml 의존성 30~50줄 복붙 ④ web.xml, servlet-context.xml, root-context.xml 작성 ⑤ application.properties (DB 정보) — 또는 root-context.xml 안에 직접 ⑥ log4j 설정 ⑦ 한 번 실행 — 안 뜨면 7개 파일 중 어디가 어긋났는지 찾기

3. SpringBoot — 등장 배경

"매번 같은 설정을 다시 쓰지 맙시다"

Spring 팀이 같은 고민을 했습니다. 99% 같은 설정이라면 — 기본값을 프레임워크가 미리 들고 있으면 되지 않을까? 사용자는 다른 1% 만 적으면 되도록. 이게 SpringBoot 의 핵심 철학입니다.

이미 조립된 베이스 키트 위에서 출발하는 셈입니다. Spring 자체가 사라진 게 아니라, 설정만 미리 깔아둔 Spring 입니다. 우리가 익힌 IoC, DI, MVC, Controller, Service, Mapper — 모두 그대로 살아있습니다.

4. Auto-Configuration — 자동 설정의 정체

SpringBoot 의 핵심 기능입니다. 한 줄로 요약하면 — 「클래스패스에 무엇이 있는지를 보고, 필요한 빈을 알아서 등록」 합니다.

클래스패스에 spring-webmvc 가 있네? → DispatcherServlet 자동 등록 (Legacy 의 web.xml 자리) → ViewResolver 기본값 등록 (Legacy 의 servlet-context.xml 자리) 클래스패스에 tomcat-embed-core 가 있네? → 임베디드 톰캣 자동 시작 (Legacy 의 톰캣 설치 자리) 클래스패스에 mybatis-spring-boot-starter 가 있네? → SqlSessionFactory 자동 구성 (Legacy 의 root-context.xml 자리) 클래스패스에 spring-boot-starter-jdbc 가 있네? → application.properties 의 DB 정보를 읽어 DataSource 자동 등록

이 동작의 진입점이 @SpringBootApplication 어노테이션 한 줄입니다. 사실 이건 세 어노테이션의 묶음입니다 — @Configuration, @EnableAutoConfiguration, @ComponentScan. 학생은 세 개를 외울 필요 없이 한 단어만 기억하면 됩니다.

「조건부」 자동 설정

SpringBoot 의 자동 설정은 무조건 다 켜는 게 아닙니다. @ConditionalOnClass, @ConditionalOnMissingBean 같은 조건 어노테이션이 붙어있어서 — 클래스패스에 라이브러리가 있을 때만, 또는 사용자가 직접 빈을 등록하지 않았을 때만 자동 등록합니다.

그래서 「내가 직접 정의하면 그게 우선」 입니다. 자동 설정은 빈자리를 메우는 역할.

4-1. @SpringBootApplication 의 정체

이 한 줄 어노테이션을 풀면 다음 셋의 묶음입니다:

@SpringBootApplication
↓ 풀어보면
@Configuration            // 이 클래스가 빈 정의의 근원지
@EnableAutoConfiguration  // 자동 설정 메커니즘 켜기
@ComponentScan            // 같은 패키지 이하의 @Component 스캔

이 중 핵심은 @EnableAutoConfiguration — 부팅할 때 SpringBoot 의 jar 안에 들어 있는 spring.factories (또는 최신 버전의 AutoConfiguration.imports) 파일을 읽고, 그 안에 적힌 모든 *AutoConfiguration 클래스를 후보로 올립니다. 그 후보 각각이 자기 조건을 검사해서 — 통과하면 빈을 등록합니다.

4-2. 디버깅 — 무엇이 자동 설정됐는지 보고 싶다

SpringBoot 가 「자동으로」 한 일을 눈으로 보고 싶을 때 — application.properties 에 한 줄:

debug=true

그러면 콘솔에 CONDITIONS EVALUATION REPORT 가 찍힙니다. Positive matches 섹션이 「조건이 맞아서 자동 설정된 빈들」, Negative matches 섹션이 「조건이 안 맞아서 건너뛴 빈들」. 본 과정의 디버깅 사고와 정확히 같은 패턴 — 「데이터 흐름 어디까지 갔는지」 를 로그로 추적합니다.

5. Starter — 묶음 의존성

Legacy 시절 가장 큰 통증 중 하나가 버전 충돌 이었습니다. spring-webmvc 5.3 에는 어떤 jackson 버전이 호환되는지, mybatis-spring 은 어느 버전인지 — 사용자가 다 챙겨야 했습니다.

spring-boot-starter-web 한 줄

웹 애플리케이션에 필요한 의존성을 「이미 검증된 버전으로 묶어둔 세트」 입니다.

spring-boot-starter-web 한 줄 추가하면 ├── spring-webmvc (호환 버전 자동 선택) ├── spring-web (호환 버전 자동 선택) ├── tomcat-embed-core (임베디드 톰캣) ├── jackson-databind (JSON 변환) ├── spring-boot-starter (코어) └── spring-boot-starter-logging

버전 번호를 우리가 안 적습니다. SpringBoot 의 부모 POM 한 곳 이 모든 하위 의존성의 호환 버전을 결정합니다. 「mybatis-spring 2.0.6 으로 올렸더니 spring 이 안 맞네…」 같은 야근이 사라집니다.

주요 Starter — 이름만 봐도 어떤 일을 하는지 짐작이 갑니다:

  • spring-boot-starter-web — REST/MVC 웹 애플리케이션
  • spring-boot-starter-jdbc — JDBC + HikariCP 커넥션 풀
  • mybatis-spring-boot-starter — MyBatis 자동 구성 (서드파티 제공)
  • spring-boot-starter-security — Spring Security
  • spring-boot-starter-test — JUnit + Mockito 등 테스트 도구

6. 임베디드 톰캣 — main() 으로 바로 실행

Legacy 에서는 톰캣을 따로 설치 → war 를 톰캣에 배포 했습니다. SpringBoot 는 톰캣을 jar 안에 품고 있어서 — java -jar app.jar 한 줄로 실행됩니다.

[Legacy 실행] ① 톰캣 설치 ② Eclipse 에 Server 등록 ③ 프로젝트를 Server 에 Add ④ Server Start ⑤ http://localhost:8080/myapp/hello [Boot 실행] ① main() 우클릭 → Run ② http://localhost:8080/hello

학생이 첫날 만나는 「환경 세팅의 벽」 자체가 사라집니다. IDE 가 없어도 됩니다 — jar 파일만 있으면 어떤 머신에서든 실행됩니다. 도커 이미지로 만들기도 쉽고, 클라우드 배포도 단순해집니다(이 부분은 후속 과정).

7. Hello World — 코드 비교

같은 「/hello 에 접속하면 인사말」 화면을 두 방식으로 비교합니다.

Legacy

web.xml + servlet-context.xml + root-context.xml + pom.xml + HelloController.java + hello.jsp + 톰캣 설치/등록

파일 6개 + 환경 세팅. 처음 화면 띄우는 데 30분~1시간.

→
SpringBoot

Application.java (10줄) + HelloController.java + application.properties (선택) + pom.xml 의 starter 한 줄

파일 3개. 처음 화면 띄우는 데 5분.

Application.java — 단 10줄
package com.example.demo;

import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;

@SpringBootApplication
public class DemoApplication {
    public static void main(String[] args) {
        SpringApplication.run(DemoApplication.class, args);
    }
}

이 한 파일이 — Legacy 의 web.xml + servlet-context.xml + root-context.xml + 톰캣 설정을 전부 대체합니다.

HelloController.java — Legacy 와 100% 동일
@Controller
public class HelloController {
    @GetMapping("/hello")
    public String hello(Model model) {
        model.addAttribute("msg", "안녕, SpringBoot!");
        return "hello";
    }
}

본 과정에서 익힌 @Controller, @GetMapping, Model — 모두 그대로 살아있습니다. 바뀐 건 설정의 무게뿐, 코어 사용법은 동일.

8. application.properties — 7개 XML 이 한 파일로

# 서버 포트 (Legacy 의 Tomcat server.xml 자리)
server.port=8080

# DB 연결 (Legacy 의 root-context.xml 자리)
spring.datasource.url=jdbc:mysql://localhost:3306/myapp
spring.datasource.username=root
spring.datasource.password=secret

# JSP 위치 (Legacy 의 ViewResolver 설정 자리)
spring.mvc.view.prefix=/WEB-INF/views/
spring.mvc.view.suffix=.jsp

# MyBatis (Legacy 의 SqlSessionFactory 자리)
mybatis.mapper-locations=classpath:mappers/*.xml
mybatis.configuration.map-underscore-to-camel-case=true

키 = 값 형식의 평범한 텍스트입니다. XML 의 중첩 태그가 사라졌고, 키 이름만 봐도 무엇을 설정하는지 짐작이 갑니다.

9. SpringBoot 가 안 풀어주는 것

착각하기 쉬운 부분

SpringBoot 는 설정의 무게 만 줄여줍니다. 다음은 여전히 우리 책임입니다:

  • 비즈니스 로직 — 게시판 권한 체크, 검증 규칙, 도메인 흐름
  • DB 스키마 설계 — 어떤 테이블, 어떤 컬럼, 어떤 인덱스
  • SQL — MyBatis 쿼리는 직접 작성
  • 화면 흐름 — Controller → Service → Mapper 4 박스 사고는 그대로
  • 디버깅 사고 — 흐름도 역추적, F12, 콘솔 로그, SQL 로그 — 동일

본 과정에서 익힌 「데이터 흐름」 사고는 100% 그대로 살아있습니다. 오히려 더 잘 보입니다 — 설정 잡음이 줄었기 때문에.

10. Before / After — 한 화면

Spring Legacy
  • web.xml ~100 줄
  • servlet-context.xml ~60 줄
  • root-context.xml ~40 줄
  • pom.xml 의존성 ~50 줄
  • 톰캣 별도 설치 / war 배포
  • 버전 충돌 직접 해결
  • Controller / Service / Mapper 직접 작성
→
SpringBoot
  • web.xml — 0 줄
  • application.properties 1 개
  • pom.xml 의존성 ~5 줄 (starter)
  • 임베디드 톰캣 (별도 설치 X)
  • java -jar 로 실행
  • 버전은 SpringBoot 가 결정
  • Controller / Service / Mapper 그대로

11. 다음에 무엇이 오는가

오늘은 「개념 둘러보기」 까지입니다. 후속 과정에서 손으로 만질 주제들:

  • Spring Initializr 로 첫 프로젝트 — 5분 안에 Hello
  • JPA — MyBatis 대신 객체로 DB 다루기
  • JWT 인증 — 세션 대신 토큰 기반 로그인
  • Docker 배포 — jar 를 컨테이너로 띄우기
  • Spring Security — 인터셉터 대신 표준 보안 프레임워크

본 과정의 v∞ 가 종착점이 아니라, 다음 과정의 출발점이 됩니다.

12. 오늘 들고 가는 3가지

  • SpringBoot = 설정이 미리 깔린 Spring. 코어는 그대로, 무게만 덜어냄.
  • @SpringBootApplication + Starter 두 단어가 핵심 — 자동 설정 + 묶음 의존성.
  • 비즈니스 로직과 흐름 사고는 여전히 우리 책임 — 본 과정의 4 박스 사고가 그대로 살아있음.
