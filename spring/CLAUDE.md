## 프로젝트 개요

**Spring 기초 과정 웹사이트**입니다. [`COURSE-TEMPLATE-GUIDE.md`](./COURSE-TEMPLATE-GUIDE.md)의 템플릿 구조와 [`자료/`](./자료) 디렉토리의 8개 docx 자료를 기반으로 합니다. 마무리 산출물은 **REST API + Ajax 게시판**이며, JPA·JWT·Docker 같은 심화 주제는 후속 과정으로 분리되었습니다.

**`COURSE-TEMPLATE-GUIDE.md`를 디자인 스펙으로, `자료/` 디렉토리를 콘텐츠 스펙으로** 취급하세요.

> ⚠️ **표기 원칙**: 학습자(수강생)가 보는 어떤 화면에도 「**비전공자**」라는 단어를 노출하지 않는다. 이 과정의 대상이 비전공자라는 것은 **강사의 내부 포지셔닝일 뿐**이며 학생용 산출물에는 등장하면 안 된다. 코스명·hero·헤더·footer·핸드아웃 표지·학습 목표·체크리스트 어디에도 사용 금지.

## 핵심 교수법: 이터레이션(반복 개선) 패턴

이 과정의 모든 콘텐츠는 **하나의 프로젝트가 v0 → v∞ 로 진화하는 여정**으로 짜여 있습니다. 차시는 독립적인 토픽이 아니라 **"이번 시간 우리 프로젝트가 어디로 진화하는가"** 의 한 단계입니다.

### 버전 진화 지도

| 버전 | 마일스톤 | 위치 |
|------|---------|------|
| v0   | Hello Servlet | Part 1 |
| v0.5 | DB 없이 종단간 흐름 (서비스에서 하드코딩) | Part 3 |
| v1   | DB 첫 연동 — 회원 한 줄 꺼내기 | Part 4 |
| v2   | 회원가입/로그인 v1 — "원시적으로" (평문 / 세션 없음) | Part 5 |
| v3   | 회원가입/로그인 v2 — "안전하게" (BCrypt + 세션) | Part 5 |
| v4   | 로그인 v3 — "깔끔하게" (인터셉터 + 공통 레이아웃) | Part 5 |
| v5   | 게시판 v1 — "최소" (Create + Read, 인가 없음) | Part 5 |
| v6   | 게시판 v2 — "안전" (Update/Delete + 본인 확인) | Part 5 |
| v7   | 게시판 v3 — 페이징 (보강) | Part 5 보강 |

| v∞   | 디버깅 워크숍 — 흐름도 역추적으로 마무리 | Part 6 |

## 모든 차시는 4단계 구조를 따른다 (필수)

차시 콘텐츠(slides, examples, labs)는 항상 다음 4 단계 흐름으로 작성합니다:

1. **⚠️ 이전 버전의 불편** (`.pain-point`) — "어디가 아팠는가"를 학생이 직접 체감하게 함
2. **🛠️ 새 도구 소개** (`.new-tool`) — 비유 + 어떤 점을 풀어주는가
3. **💻 코드 적용** — 같은 화면을 다시 만든다
4. **🔄 Before / After 비교** (`.before-after`) — "데이터 흐름이 어디가 달라졌나"

차시가 아무리 작아도 이 네 박자 중 **불편 → 적용 → Before/After 세 가지는 빠지지 않습니다.** Part 1·2 의 도입부 차시처럼 "이전 버전이 없는" 경우에는 ① 자리에 **"왜 이게 필요한가"** 를 둡니다.

## 데이터 흐름 컨벤션 3종 (필수)

학생이 디버깅 사고를 자연스럽게 익히도록 모든 차시 자료에 다음 3가지를 표준으로 박습니다.

### 1. 흐름도(Flow Diagram) 슬라이드 — 슬라이드 마지막 직전에 고정 위치
```
[브라우저] → [Controller] → [Service] → [Mapper] → [DB]
                                    ↑
                              이번 차시에서 새로 생긴 칸
```
- `.flow-diagram` 클래스 사용
- 차시가 누적될수록 흐름도가 길어지고, 새로 생긴 칸은 강조색으로 표시
- 학생은 매 차시 끝에 "이번 시간에 어디가 새로 생겼는지" 한 화면으로 본다

### 2. 미니맵 — examples / labs 첫 페이지에 고정
- 전체 흐름도 중 이번 차시가 닿는 부분을 색으로 강조
- 코드를 만지기 전에 **"내가 지금 어디를 만지는가"** 를 먼저 인식

### 3. 확인 포인트(Checkpoint) — labs 의 모든 Step 마지막에 고정
- `.checkpoint` 클래스 사용
- 콘솔 로그 / F12 네트워크 탭 / MyBatis SQL 로그 / DB SELECT 중 어느 것을 볼지 매번 명시
- 단계마다 **"내 데이터가 여기까지 왔는지"** 를 직접 확인하는 습관을 기른다

## 차시 식별: 숫자 대신 기호 (절대 차시 번호 노출 금지)

학생에게 "지금 23/64 차시" 같은 압박을 주지 않습니다. 차시는 다음 기호로만 표시:

| 기호 | 의미 |
|------|------|
| ◇ | 개념 차시 (concept) |
| ▣ | 실습 차시 (hands-on) |
| ◆ | 전환점 (새 도구 / 패러다임이 처음 등장) |
| ★ | 마일스톤 (프로젝트 새 버전 v_n_ 릴리스) |

UI 라벨 형식:
- 일반: `◇ 카페 비유로 시작` / `▣ Hello Servlet 띄우기`
- 마일스톤: `★ v3 — 안전한 로그인`

**파일명**은 의미 기반(Part 약어 + 키워드)으로 둡니다. 학습자에게 보이지 않습니다:
- `slides/web-network-cafe.html`
- `slides/spring-ioc-taxi.html`
- `slides/auth-v2-raw.html`
- `slides/auth-v3-secure.html`
- `slides/board-v5-minimal.html`
- `slides/rest-v8-conversion.html`

Part 약어 표:

| Part | 약어 | 색상 클래스 |
|------|------|-----------|
| 웹의 동작 원리 | `web` | `.part-1` 파랑 |
| Java와 Spring 철학 | `spring` | `.part-2` 초록 |
| Spring MVC 아키텍처 | `mvc` | `.part-3` 보라 |
| 데이터와 MyBatis | `db` | `.part-4` 오렌지 |
| 회원과 게시판 | `auth` / `board` | `.part-5` 빨강 |
| REST API와 마무리 | `rest` | `.part-6` 틸 |
| 팀 분업 시뮬레이션 (◆ 전환점) | `team` | (회색) |

## 아키텍처 개요 (자료 유형 4종)

각 세션은 **네 가지 산출물**을 형제 디렉토리에 둡니다 (가이드 §2). 파일명은 의미 기반:

| 디렉토리 | 용도 | 테마 | 프레임워크 |
|----------|------|------|-----------|
| `slides/` | 발표 (4단계 구조 + 흐름도 슬라이드) | 다크 | reveal.js 5.1.0 (night) |
| `examples/` | 읽기 자료 (개념 + 미니맵 + 비유) | 다크 | 순수 HTML |
| `labs/` | 단계별 실습 (Step + Checkpoint) | 다크 | 순수 HTML |
| `handouts/` | 인쇄용 요약 (3~5쪽) | 라이트 | 순수 HTML (`@media print`) |

`index.html`은 6 Part × N 마일스톤·일반 카드. `js/screenshot.js`는 모든 자료에서 공유하는 스크린샷 런타임 (가이드 §7).

### 🚨 비유 사용 원칙 (가장 중요)

**비유는 본문 안에서 학생의 이해를 돕는 도구로만 사용**합니다. 다음에는 절대 등장하면 안 됩니다:

- ❌ 코스 이름 / 부제 (예: ~~"고급 식당으로 배우는 게시판"~~)
- ❌ Part 라벨 / Part tagline (예: ~~「손님과 가게」, 「설계도와 주입」, 「식재료 창고와 통역사」~~)
- ❌ 카드 제목 (예: ~~"카페 비유로 시작", "기억상실증 점원 문제"~~)
- ❌ 슬라이드 / 예제 / 실습 / 핸드아웃의 h1·h2·h3 섹션 제목
- ❌ 학습 목표 항목 (예: ~~"카페 비유로 설명할 수 있다"~~ → "자기 말로 설명할 수 있다")
- ❌ 체크리스트 항목
- ❌ 실습 Step 제목

**대신 비유는 다음 자리에서만**:

- ✅ 본문 단락 안의 보조 설명 (예: "서버는 카페의 바리스타에 해당합니다")
- ✅ `.diagram` 안의 텍스트 다이어그램
- ✅ `.new-tool` / `.box` 안의 풀어쓰기
- ✅ 슬라이드의 강사 노트
- ✅ 핸드아웃의 「용어 ↔ 비유」 매핑 표 (표 자체가 매핑이 목적인 경우)

### 핵심 비유 사전 (본문에서 일관되게 사용)
  - 서버 / 클라이언트 → 카페의 **바리스타 / 손님**
  - HTTP 무상태성 → **기억상실증 점원**
  - IoC → **택시 뒷좌석**
  - DI → **재료 배달**
  - 클래스 vs 인스턴스 → **설계도 vs 제품**
  - DispatcherServlet → **안내데스크**
  - Controller → **종업원·지배인**
  - Service → **메인 셰프**
  - DAO/Repository → **창고 관리자**
  - DTO/VO → **그릇**
  - View → **식탁·메뉴판**
  - DB → **엑셀 시트**
  - MyBatis → **자바 ↔ SQL 통역사**
  - HikariCP 커넥션 풀 → **미리 깎아둔 칼통**
  - 세션 → **손목 도장**
  - 쿠키 → **영수증** (저장 위치가 손님 쪽)
  - BCrypt → **단방향 자물쇠** (서버도 진짜 비밀번호를 모름)
  - Ajax → **몰래 다녀오는 통신원**

- **셸 코드 블록**: 학생 입력은 `$ ` 접두사, 출력은 접두사 없음.
- **하단 네비게이션 필수**: 모든 세션 페이지 하단에 `.nav` (이전 / 메인 / 다음).

## 새 차시를 만들 때

1. 해당 차시가 **어떤 버전 진화에 속하는지** 먼저 정한다 (v_n_ → v_(n+1)_)
2. **4 단계 구조**로 본문을 짠다 (불편 → 도구 → 적용 → Before/After)
3. 슬라이드 마지막 직전에 **흐름도 슬라이드**를 둔다 (이번에 새로 생긴 칸 강조)
4. examples / labs 첫 페이지에 **미니맵**을 둔다
5. labs 의 모든 Step 마지막에 **Checkpoint** 를 둔다
6. 마일스톤 차시면 헤더에 **버전 뱃지** 를 단다
7. `index.html` 의 해당 Part 섹션에 **카드 한 장** 을 추가한다 (기호 + 키워드, 차시 번호 없음)

- ❌ 비유를 **제목·라벨·학습 목표·체크리스트**로 사용 (본문 보조 설명 자리에서만 OK)

- ❌ 흐름도 슬라이드 없이 차시 마무리

- ❌ JPA / JWT / Docker / SOLID / 전역예외 같은 후속 과정 주제 도입
