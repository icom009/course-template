2. Hello World — 두 파일이면 끝

Application.java (시작점 — 단 10줄)

package com.example.demo; import org.springframework.boot.SpringApplication; import org.springframework.boot.autoconfigure.SpringBootApplication; @SpringBootApplication public class DemoApplication { public static void main(String[] args) { SpringApplication.run(DemoApplication.class, args); } }

HelloController.java (Legacy 와 동일)

package com.example.demo; import org.springframework.web.bind.annotation.GetMapping; import org.springframework.web.bind.annotation.RestController; @RestController public class HelloController { @GetMapping("/hello") public String hello() { return "안녕, SpringBoot!"; } }

pom.xml (의존성 한 줄)

<parent> <groupId>org.springframework.boot</groupId> <artifactId>spring-boot-starter-parent</artifactId> <version>3.2.0</version> </parent> <dependencies> <dependency> <groupId>org.springframework.boot</groupId> <artifactId>spring-boot-starter-web</artifactId> </dependency> </dependencies>

용어비유 / 한 줄 풀이
SpringBoot이미 조립된 베이스 키트 — 설정만 미리 깔린 Spring
@SpringBootApplication「자동 설정 / 컴포넌트 스캔 / Configuration」 세 어노테이션을 묶은 한 단어
Auto-Configuration클래스패스에 무엇이 있는지 보고 — 빈자리를 알아서 메우는 메커니즘
Starter호환 검증된 의존성 묶음 — 「웹 한 세트」, 「JDBC 한 세트」

@RestController본 과정의 @Controller + @ResponseBody — JSP 없이 본문 텍스트로 응답

4. SpringBoot 가 안 풀어주는 것

Boot 는 설정의 무게 만 줄여줍니다. 다음은 여전히 우리 몫입니다:
  • 비즈니스 로직 — 게시판 권한 체크, 검증 규칙, 도메인 흐름
  • DB 스키마 설계 — 어떤 테이블, 어떤 컬럼, 어떤 인덱스
  • SQL — MyBatis 쿼리는 직접 작성
  • 화면 흐름 — Controller → Service → Mapper 4 박스 사고
  • 디버깅 사고 — 흐름도 역추적, F12, 콘솔, SQL 로그

5. 다음 과정에서 만나게 될 것

주제본 과정과의 관계
Spring Initializrstart.spring.io 에서 5분 안에 첫 Boot 프로젝트 생성. 본 실습의 pom.xml 작성을 자동화.
JPA / Spring Data JPAMyBatis 대신 객체로 DB 다루기. 본 과정의 DAO/Mapper 사고를 객체 그래프로 확장.
JWT 인증세션 (손목 도장) 대신 토큰 기반 로그인. 본 과정의 v3 안전한 로그인의 다음 진화.
Spring Security인터셉터 + 본인 확인 (v4) 대신 표준 보안 프레임워크.
Docker 배포jar 를 컨테이너 이미지로 만들어 어디서든 띄우기.
Spring Boot Actuator실행 중인 앱의 상태 (메모리, 헬스 체크) 를 표준 엔드포인트로 노출.
학습 전략의 의미: 본 과정에서 Legacy 로 시작한 이유 — XML 한 줄 한 줄을 직접 짜며 「데이터 흐름이 어디서 끊겼는지」 를 짚을 수 있게 하기 위해서였습니다. 후속 과정에서 Boot 로 옮길 때, 자동 설정이 무엇을 자동화했는지를 「되긴 하는데 왜 되는지 모르는」 상태가 아니라 손에 잡히는 상태로 만나게 됩니다.

6. 본 과정에서 익힌 것 — 그대로 쓸 수 있는 것

본 과정에서 익힌 것SpringBoot 에서의 위치
HTTP 무상태성 / GET / POST그대로 — 프로토콜 이해는 프레임워크와 무관
IoC / DI / @Autowired그대로 — Spring 의 코어 그 자체
Controller → Service → DAO 4 박스 사고그대로 — Boot 가 바꾸지 않음
MyBatis Mapper XML / SQL그대로 — starter 가 자동 구성만 추가
세션 / 쿠키 / BCrypt그대로 — 후속 과정에서 JWT 로 대체 가능
F12 / 콘솔 로그 / SQL 로그 디버깅그대로 — 오히려 설정 잡음이 줄어 더 잘 보임

학습 확인 체크리스트

  • SpringBoot 가 Spring 위에 얹힌 「자동 설정 도우미」 라는 사실을 안다
  • @SpringBootApplication 한 줄이 어떤 일을 하는지 안다
  • Starter 의존성이 버전 충돌을 어떻게 해결하는지 안다
  • 임베디드 톰캣이 무엇을 바꿨는지 안다
  • SpringBoot 가 안 풀어주는 것 (비즈니스 로직, SQL, 흐름 사고) 을 구분할 수 있다
  • 본 과정의 v∞ 가 다음 과정의 출발점이 된다는 사실에 동의한다
한 줄 정리: SpringBoot 는 「설정의 무게만 덜어낸 Spring」 입니다. 본 과정에서 익힌 데이터 흐름·MVC·MyBatis·세션·디버깅 사고는 그대로 살아 있고, XML 7개를 매번 다시 쓰던 부담만 사라집니다. 후속 과정에서 손으로 만나게 됩니다.
