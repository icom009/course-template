클라이언트와 서버

웹의 동작 원리 — 요청과 응답

첫 시간이므로 분위기를 편하게 잡아주세요. 컴퓨터에 익숙하지 않아도 괜찮다고 안내합니다. 오늘 우리는 코드를 한 줄도 쓰지 않습니다. 매일 보는 웹사이트가 어떻게 동작하는지를 누구나 아는 풍경(카페에서 커피 시키기)에 빗대어 짚어봅니다.

학습 목표

  • 클라이언트와 서버가 무엇인지 자기 말로 설명할 수 있다
  • 웹사이트에서 일어나는 요청과 응답의 흐름을 그릴 수 있다
  • 실제 브라우저에서 이 흐름이 일어나는 순간을 직접 확인할 수 있다
학습 목표를 읽어주고, 끝나고 나서 학생이 스스로 "할 수 있게 됐는지" 확인할 거라고 안내.

⚠️ 왜 이걸 알아야 할까요?

우리가 매일 만나는 풍경

주소창에 naver.com 을 치면 — 1초 만에 화면이 뜹니다.
그 1초 동안 컴퓨터 사이에서는 무슨 일이 일어났을까요?

이걸 모르면 나중에 코드가 안 될 때 어디서 막혔는지 짐작할 수가 없습니다.

입문자에게 가장 와닿는 도입은 "이거 모르면 디버깅이 안 된다"입니다. 코드를 짤 줄 아는 것과, 안 될 때 어디를 봐야 할지 아는 것은 다른 기술이라고 짚어주세요.

🛠️ 카페에서 일어나는 일

웹 = 카페

웹사이트의 모든 동작은 카페에서 커피를 시키는 것과 똑같이 생겼습니다.

┌─────────────┐ ┌──────────────┐ │ │ "아메리카노" │ │ │ 손님 │ ──────────────→ │ 바리스타 │ │ (Client) │ │ (Server) │ │ │ ←────────────── │ │ │ │ ☕ 한 잔 │ │ └─────────────┘ └──────────────┘
여기가 핵심 비유입니다. 한 번 잡히면 4주 내내 이 그림이 머릿속에 박힙니다. 학생들에게 "여러분도 매일 이걸 하고 있어요"라고 친근하게.

손님 — 클라이언트 (Client)

  • 먼저 "이거 주세요" 라고 말을 거는 쪽
  • 우리가 매일 쓰는 크롬, 사파리
  • 배달 앱, 게임, 쇼핑 앱도 전부 클라이언트
💡 한 줄 정리

"먼저 부탁하는 쪽" 이면 모두 클라이언트.

학생들이 매일 쓰는 앱이 전부 클라이언트라는 걸 짚어주면 친근해집니다.

바리스타 — 서버 (Server)

  • 손님의 요청을 묵묵히 기다리는 쪽
  • 언제 손님이 올지 모르므로 24시간 켜져 있어야 함
  • 네이버 서버, 유튜브 서버, 카카오톡 서버 ...
⏰ 중요한 차이

서버는 먼저 말을 걸지 않습니다.
항상 손님이 먼저입니다.

서버가 24시간 켜져 있다는 점, 그리고 절대 먼저 말을 걸지 않는다는 점이 학생들에게는 의외의 사실입니다.

요청과 응답 (Request / Response)

1. 손님 → 가게: "아메리카노 한 잔 주세요" ← 요청 (Request) 2. 가게 → 손님: "여기 있습니다 ☕" ← 응답 (Response)

웹에서 우리가 하는 모든 일은 이 두 단계의 반복입니다.

🎯 외워둘 것

웹 = 요청(Request) + 응답(Response). 끝.

이 두 단어가 4주 내내 등장합니다. 지금 단단히 박아두면 나중에 편합니다.

네이버 화면이 뜨는 그 1초

주소창에 naver.com 입력 │ ▼ ┌─────────────┐ ┌──────────────┐ │ 브라우저 │ "네이버 줘" │ 네이버 서버 │ │ (크롬) │ ─────────────────→ │ │ │ │ │ │ │ │ ←───────────────── │ │ │ │ HTML, 이미지 등 │ │ └─────────────┘ └──────────────┘ │ ▼ 화면을 그려서 보여줌
이 1초 안에 일어나는 일을 그림으로 보여주면 학생들이 "아, 이렇게 되는 거구나" 합니다. 이게 안 되면 "인터넷이 안 돼요"가 되는 거죠.

💻 진짜로 일어나는지 보고 싶다면

크롬을 열고 — F12 를 눌러보세요.

스크린샷
크롬에서 네이버에 접속한 뒤 F12 → Network 탭. 수십 개의 요청·응답이 줄지어 보임

한 페이지를 띄우는 데 사실은 수십 번의 요청·응답이 오갑니다.

여기서 학생들이 처음으로 F12 를 누르는 경험을 합니다. 두려워할 필요 없다고 말해주세요. 실습 자료(labs/)에서 단계별로 따라하게 됩니다.

📊 한 그림 정리

이번 차시의 데이터 흐름

브라우저
⇄
서버
● 첫 박스 두 개가 등장했습니다 — 앞으로 이 사이가 점점 채워집니다

오늘은 그저 둘이 대화하는 사이임을 봤습니다.
다음 차시부터 사이에 박스가 하나씩 추가됩니다.

이 흐름도는 매 차시마다 점점 길어집니다. 박스가 하나씩 채워지는 게 우리 과정의 진도입니다. 마지막 차시 즈음에는 [브라우저] → [Controller] → [Service] → [Mapper] → [DB] 까지 늘어납니다.

정리

오늘 들고 가는 것

  • 클라이언트 = 먼저 부탁하는 쪽 (손님, 브라우저)
  • 서버 = 묵묵히 기다리는 쪽 (바리스타, 24시간 켜진 컴퓨터)
  • 웹의 모든 일 = 요청 + 응답의 반복

다음 차시: HTTP 해부 — 요청·응답 안에 어떤 글자가 들어있는지 F12 로 직접 뜯어봅니다.

정리는 짧게. 학생이 "오늘은 이걸 들고 갑니다"를 입으로 말할 수 있게.
