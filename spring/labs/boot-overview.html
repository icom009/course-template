Legacy 에서 30분 걸리던 Hello 를 — 같은 결과로 5분 안에 띄워봅니다. 「설정의 무게」 가 빠졌을 때 무엇이 달라지는지 손으로 확인.

사전 준비

  • JDK 17 이상 (Boot 3.x 기준) 또는 JDK 11 (Boot 2.7.x)
  • Eclipse / STS 또는 IntelliJ — 본 실습은 Eclipse 기준
  • Maven (Eclipse 내장)
  • 인터넷 연결 (의존성 다운로드용)
  • 본 과정에서 만든 Legacy Hello 프로젝트 — 비교용으로 옆에 열어두면 좋음
실습 전 마음가짐

본 과정에서 Legacy Hello 를 띄우는 데 — 처음에는 30분~1시간이 걸렸을 겁니다. 톰캣 등록, web.xml, servlet-context.xml, ViewResolver, JSP 위치 ··· 이번에는 같은 결과를 5분 안에 띄워봅니다. 그 차이가 어디서 오는지 — Step 마다 콘솔과 코드를 비교하며 직접 확인하세요.

이번 실습의 목표

① spring-boot-starter-web 한 줄을 추가하고 ② @SpringBootApplication 클래스를 만들어 ③ main() 으로 톰캣을 띄우고 ④ HelloController 로 화면을 받아냅니다. Legacy 에서 만지던 web.xml / servlet-context.xml / root-context.xml 은 만지지 않습니다.

1
pom.xml 에 starter-web 한 줄

Eclipse 에서 새 Maven 프로젝트를 만들고 pom.xml 을 다음과 같이 작성합니다. parent 설정과 starter 의존성 한 줄 이 핵심.

<project xmlns="http://maven.apache.org/POM/4.0.0">
  <modelVersion>4.0.0</modelVersion>

  <groupId>com.example</groupId>
  <artifactId>demo</artifactId>
  <version>0.0.1</version>

  <parent>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-parent</artifactId>
    <version>3.2.0</version>
  </parent>

  <dependencies>
    <dependency>
      <groupId>org.springframework.boot</groupId>
      <artifactId>spring-boot-starter-web</artifactId>
    </dependency>
  </dependencies>
</project>

버전 번호가 한 곳에만 있습니다 — parent 의 3.2.0. starter-web 자체에는 버전이 없습니다. parent 가 결정합니다.

참고 — Legacy 의 같은 자리는 어땠나
<!-- Legacy pom.xml — 의존성 한 묶음의 일부 -->
<dependency>
  <groupId>org.springframework</groupId>
  <artifactId>spring-webmvc</artifactId>
  <version>5.3.20</version>
</dependency>
<dependency>
  <groupId>javax.servlet</groupId>
  <artifactId>jstl</artifactId>
  <version>1.2</version>
</dependency>
<!-- ... 30~50줄 더 ... -->

Boot 의 starter 한 줄이 — Legacy 의 30~50줄을 대체합니다. 게다가 버전 충돌도 사라집니다.

CHECKPOINT
  • 저장 후 Maven 이 의존성을 다운로드하는지 — Eclipse 좌측 하단 진행 표시 확인.
  • 프로젝트의 Maven Dependencies 폴더를 열어 tomcat-embed-core, spring-webmvc, jackson-databind 가 들어왔는지 눈으로 확인. starter 한 줄이 이만큼을 가져옴.
  • 의존성 트리를 자세히 보고 싶다면 — 터미널에서 mvn dependency:tree. starter-web 아래에 무엇이 매달려 있는지 한 눈에.
  • 빨간 X 표시가 없으면 통과. 있다면 인터넷 연결 / Maven 저장소 설정 점검.
2
Application.java — 시작점 클래스

src/main/java/com/example/demo/ 아래에 DemoApplication.java 작성:

package com.example.demo;

import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;

@SpringBootApplication
public class DemoApplication {
    public static void main(String[] args) {
        SpringApplication.run(DemoApplication.class, args);
    }
}

이 10줄짜리 파일이 — Legacy 의 web.xml + servlet-context.xml + root-context.xml + 톰캣 등록을 전부 대체합니다. @SpringBootApplication 한 줄이 자동 설정 + 컴포넌트 스캔 + Configuration 을 모두 켭니다.

패키지 위치가 「루트」 여야 하는 이유

@SpringBootApplication 안에는 @ComponentScan 이 들어 있고, 이 스캔의 시작점은 이 클래스가 속한 패키지 (그리고 그 하위) 입니다. 컨트롤러·서비스를 com.example.demo 와 동일하거나 그 아래 (예: com.example.demo.controller) 에 둬야 잡힙니다. 위로(예: com.example.other) 두면 못 찾아서 404.

CHECKPOINT
  • Eclipse 가 SpringApplication, @SpringBootApplication import 를 자동으로 잡아주는지 확인. 빨간 줄 없음.
  • 패키지 구조가 com.example.demo 인지 확인 — 이 패키지 아래 가 컴포넌트 스캔 범위. 컨트롤러를 다른 위치에 두면 못 찾음.
  • @SpringBootApplication 에 마우스를 올리면 — 이 어노테이션이 @Configuration, @EnableAutoConfiguration, @ComponentScan 세 개를 묶은 것이라는 설명이 보임. 셋의 묶음임을 한 번 눈으로 확인.
3
실행 — main() 우클릭 → Run As → Java Application

「Run on Server」 가 아닙니다. 일반 Java 프로그램처럼 main() 을 그냥 실행 합니다. 임베디드 톰캣이 따라옵니다.

로그 한 줄씩 풀어보기
  • Tomcat initialized with port(s): 8080 — 임베디드 톰캣이 메모리에 올라옴. Legacy 의 「Server Start」 자리.
  • Starting Servlet engine — 서블릿 컨테이너가 켜짐. DispatcherServlet 이 자동 등록되는 자리.
  • Tomcat started on port(s): 8080 — 실제로 8080 포트가 listen 상태가 됨.
  • Started DemoApplication in X.XXX seconds — 모든 자동 설정이 끝나고 요청을 받을 준비. 이 줄이 안 나오면 안 뜬 것.
CHECKPOINT — 로그 확인
  • 마지막 줄에 Started DemoApplication in X.XXX seconds 가 나오는가? — 이 한 줄이 「뜨기 성공」 의 신호.
  • Tomcat started on port(s): 8080 가 보이는가? — 임베디드 톰캣이 켜진 증거.
  • 포트 충돌(Port 8080 was already in use) 이면 — 기존에 실행 중인 톰캣 종료, 또는 Step 6 에서 포트 변경.
  • 예외 스택 트레이스가 길게 떨어지면 — 「Caused by:」 줄을 끝까지 따라가서 원인 메시지 확인. 본 과정의 디버깅 사고와 동일.
4
HelloController — 화면 응답

com.example.demo 패키지 아래에 HelloController.java 작성:

package com.example.demo;

import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.RestController;

@RestController
public class HelloController {

    @GetMapping("/hello")
    public String hello() {
        return "안녕, SpringBoot!";
    }
}

@RestController 는 본 과정의 Part 6 에서 만난 @Controller + @ResponseBody 와 같은 역할입니다. 응답을 JSP 가 아니라 본문 텍스트로 바로 보냅니다.

저장하면 SpringBoot DevTools 가 있다면 자동 재시작, 없으면 Step 3 처럼 main() 을 다시 실행합니다.

Legacy 컨트롤러와 한 줄씩 비교
// Legacy
@Controller
public class HelloController {
    @GetMapping("/hello")
    @ResponseBody
    public String hello() { return "안녕"; }
}

// SpringBoot
@RestController                          // ← @Controller + @ResponseBody 묶음
public class HelloController {
    @GetMapping("/hello")
    public String hello() { return "안녕"; }
}

차이는 @RestController 한 단어. 본 과정에서 익힌 모든 어노테이션, 시그니처, 반환값 — 100% 그대로.

CHECKPOINT
  • 저장 후 빨간 X 가 없는가?
  • 콘솔에 Started DemoApplication 가 다시 보이는가?
  • 이 컨트롤러는 Legacy 의 컨트롤러와 코드가 똑같다 는 점을 확인. 어노테이션, 시그니처, 반환 — 동일.
  • Mapped 로그를 보고 싶다면 — application.properties 에 logging.level.org.springframework.web=DEBUG 한 줄. 콘솔에 Mapped "{[/hello]}" 가 나옴.
5
브라우저 확인 — localhost:8080/hello

크롬에서 http://localhost:8080/hello 접속. 화면에 「안녕, SpringBoot!」 가 보여야 합니다.

Legacy 와 비교해보면 — http://localhost:8080/myapp/hello 처럼 컨텍스트 패스가 끼지 않습니다. SpringBoot 는 기본 컨텍스트 패스가 루트 (/) 입니다.

F12 에서 무엇을 봐야 하는가
  • Headers 탭 — Request URL, Request Method (GET), Status Code (200) 가 한 화면에. 본 과정의 web-http-anatomy 에서 익힌 그대로.
  • Response 탭 — 서버가 실제로 보낸 본문. 「안녕, SpringBoot!」 텍스트가 그대로.
  • Timing 탭 — Waiting (TTFB) 이 보통 5~50ms 사이. 임베디드 톰캣은 외부 톰캣과 성능 차이 거의 없음.
CHECKPOINT — F12 Network 탭
  • F12 → Network 탭 → 페이지 새로고침. hello 요청이 보이는가?
  • Status: 200 OK 인지 확인.
  • Response Headers 의 Server: ... 또는 응답 본문이 「안녕, SpringBoot!」 인지 확인.
  • Content-Type 헤더가 text/plain;charset=UTF-8 인지 확인 — UTF-8 이 자동으로 켜진 증거 (Legacy 의 CharacterEncodingFilter 자리).
  • 404 가 뜨면 — ① 컨트롤러 패키지가 com.example.demo 아래인지, ② @RestController 가 붙어있는지, ③ main() 이 다시 실행됐는지 점검.
  • 한글이 깨져서 보인다면 — 브라우저 인코딩 설정(거의 없음) 또는 컨트롤러 반환을 produces = "text/plain;charset=UTF-8" 로 명시.
6
application.properties — 포트 변경

src/main/resources/application.properties 파일을 만들고 한 줄 추가:

server.port=9090

저장 후 main() 을 다시 실행합니다.

application.properties 에 자주 쓰는 한 줄들
# 서버 포트
server.port=9090

# 컨텍스트 패스 (Legacy 에서 /myapp 같은 자리)
server.servlet.context-path=/api

# JSON 한글 응답이 안 깨지게
server.servlet.encoding.charset=UTF-8
server.servlet.encoding.force=true

# 로그 레벨
logging.level.com.example=DEBUG
logging.level.org.springframework.web=INFO

# JSP 위치 (JSP 를 쓸 경우)
spring.mvc.view.prefix=/WEB-INF/views/
spring.mvc.view.suffix=.jsp

# DB 연결
spring.datasource.url=jdbc:mysql://localhost:3306/myapp
spring.datasource.username=root
spring.datasource.password=secret

각 줄 옆에 — Legacy 에서 어느 XML 의 어느 자리에 해당하는지 머릿속으로 매핑해보세요.

CHECKPOINT — 포트가 바뀌었는가?
  • 콘솔에 Tomcat started on port(s): 9090 (http) 가 나오는가?
  • http://localhost:9090/hello 로 접속 — 「안녕, SpringBoot!」 가 보이는가?
  • http://localhost:8080/hello 로 접속 — 이번에는 연결 거부 (이 포트로 더 이상 안 뜸).
  • 이 한 줄을 Legacy 에서 바꾸려면 — 톰캣의 server.xml 을 열어 Connector 의 port 속성을 수정해야 했음. SpringBoot 는 properties 한 줄.
  • 한 걸음 더 — debug=true 한 줄을 추가하고 다시 실행. 콘솔에 「CONDITIONS EVALUATION REPORT」 라는 긴 표가 나옴. SpringBoot 가 「자동으로」 켠 빈 목록을 직접 눈으로 볼 수 있음.

실습 완료 체크리스트

pom.xml 에 starter-web 한 줄로 webmvc + 톰캣 + jackson 이 들어왔다
@SpringBootApplication 클래스가 main() 으로 실행됐다
콘솔에서 "Started ... in X.XXX seconds" 를 확인했다
/hello 응답을 F12 Network 탭에서 200 OK 로 확인했다
application.properties 한 줄로 포트가 바뀌었다
Legacy 에서 7개 XML 로 했던 일이 어디로 갔는지 자기 말로 설명할 수 있다
데이터 흐름 — 이번 실습이 닿은 자리
[브라우저]
   ↓ HTTP GET /hello
[임베디드 톰캣 — Boot 가 자동 시작]
   ↓
[DispatcherServlet — Boot 가 자동 등록]
   ↓
[HelloController — 우리가 작성]
   ↓
[String "안녕, SpringBoot!" — 우리가 반환]
   ↓
[브라우저 화면]

본 과정의 흐름도와 정확히 같은 모양 — 단지 「설정」 박스가 사라졌을 뿐입니다.

스스로 점검 — 5분 안에 답해보세요
  1. 이 프로젝트에는 web.xml 이 없습니다. DispatcherServlet 은 어디서 등록됐을까요?
  2. spring-boot-starter-web 한 줄을 빼면 — 어떤 일이 일어날까요? (힌트: main() 실행은 되지만 톰캣이 안 뜸)
  3. HelloController 를 com.example.other 패키지에 두면 왜 404 가 날까요?
  4. Legacy 의 CharacterEncodingFilter 는 SpringBoot 의 어디로 갔을까요?
  5. application.properties 의 server.port=9090 한 줄은 — Legacy 에서 어떤 파일의 어떤 속성이었을까요?

다섯 질문에 모두 답할 수 있다면, 다음 차시(후속 과정) 에서 만날 「Spring Initializr 로 첫 프로젝트」 가 자연스럽게 이어집니다.

한 걸음 더 — 직접 비교해보기

본 과정에서 만든 Legacy 프로젝트의 web.xml, servlet-context.xml, root-context.xml 을 열어보세요. 이 중 어떤 줄이 SpringBoot 의 어떤 자동 설정으로 대체됐는지 — 한 줄씩 매핑해보면 「설정의 무게」 가 어디로 사라졌는지가 손에 잡힙니다.

  • Legacy 의 <servlet-class>DispatcherServlet</servlet-class> → Boot 의 spring-boot-starter-web 자동 등록
  • Legacy 의 InternalResourceViewResolver 빈 → Boot 의 spring.mvc.view.prefix/suffix 두 줄
  • Legacy 의 HikariDataSource 빈 → Boot 의 spring.datasource.* 네 줄
  • Legacy 의 SqlSessionFactoryBean 빈 → Boot 의 mybatis.mapper-locations 한 줄
  • Legacy 의 톰캣 외부 설치 → Boot 의 tomcat-embed-core jar (자동 포함)
자주 만나는 함정
  • 패키지 위치 — 컨트롤러를 Application 클래스보다 「위」 패키지에 두면 컴포넌트 스캔이 못 잡음. 같거나 「아래」 에 둘 것.
  • 포트 충돌 — Legacy 톰캣이 8080 으로 켜져 있는 채로 Boot 도 8080 으로 띄우면 「Port 8080 was already in use」. 둘 중 하나를 끄거나 server.port 변경.
  • JSP 가 안 보임 — Boot 에서 JSP 를 쓰려면 추가 의존성(tomcat-embed-jasper) 과 prefix/suffix 설정이 필요. 본 실습에서는 @RestController 만 사용해서 우회.
  • UTF-8 깨짐 — Boot 3.x 는 기본 UTF-8. 그래도 한글이 깨지면 IDE 의 파일 인코딩(Eclipse: Window → Preferences → General → Workspace) 부터 점검.
