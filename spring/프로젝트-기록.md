## 5. 핵심 결정 사항 — 「왜 이렇게 짰는가」

### 5-1. 이터레이션 패턴 (v0 → v∞)

전통적인 「토픽 단위 분리」 가 아니라 **하나의 프로젝트가 단계적으로 진화** 하도록 짰습니다.

```
v0    Hello Servlet
v0.5  DB 없이 종단간 흐름 (Part 3 끝)
v1    첫 DB 연동 (Part 4 끝)
v2    원시 인증 (의도적으로 평문·세션 없음)
v3    안전 인증 (BCrypt + 세션)
v4    깔끔 로그인 (인터셉터 + 공통 레이아웃)
v5    최소 게시판 (Create + Read, 인가 없음)
v6    안전 게시판 (Update/Delete + 본인 확인)
v7    페이징 게시판 (보강)
v8    REST API 컨트롤러 (View 빠진 흐름도)
v9    비동기 댓글 (Ajax)
v10   조회수·첨부파일 (REST 응용)
v∞    디버깅 워크숍 (마무리)
```

**근거**: 같은 기능을 처음에 의도적으로 어렵게 만들고, 새 도구로 다시 만들면서 **그 도구가 왜 좋은지**를 학생이 직접 체감하게 함. 자료 중 「JDBC → MyBatis Before/After」 로직을 회원/게시판/REST 전체에 확장한 형태.

### 5-2. 모든 차시 4 단계 구조

```
1. ⚠️ 이전 버전의 불편 (.pain-point)
2. 🛠️ 새 도구 소개   (.new-tool)
3. 💻 코드 적용
4. 🔄 Before/After 비교 (.before-after)
```

**근거**: 학생이 「불편을 직접 느낀 다음에 도구를 만나는」 흐름이 비전공자에게 가장 효과적.

일반 개념 차시는 Python 생성기 + 데이터셋으로 일괄 양산.

```python
# _gen.py 의 핵심 구조
SESSIONS = [
    { 'slug': '...', 'glyph': '...', 'title': '...',
      'pain': '...', 'tool_title': '...', 'tool_desc': '...',
      'code': '...', 'after': '...',
      'summary': [...], 'terms': [...] },
    # ... 39 개
]

for s in SESSIONS:
    for kind, render in [('slides', render_slides), ...]:
        path = f"{kind}/{s['slug']}.html"
        if os.path.exists(path): continue
        with open(path, 'w') as f:
            f.write(render(s))
```

`index.html` 의 모든 카드 링크를 `disabled` 에서 실제 파일 경로로 교체.

```python
# 정규식으로 일괄 활성화
pattern = re.compile(...)
content = pattern.sub(replacement, content)
```

라이브 서버 가동 후 사용자가 실제로 둘러봄. 본 프로젝트는 단계 6 후 사용자가 "내용들이 빈약하다" 지적 → 가장 중요한 9 개 슬라이드를 다시 충실히 작성.

| 3차 충실도 보강 (마일스톤 9 개) | ~150,000 | 600K |

**비고**:

- Opus 4.7 1M 컨텍스트의 약 **85 %** 사용. 한도에 닿기 전 `/compact` 1 회 발생 (4 차 시각화 시스템 작업 도중 컨텍스트 압축).
- 대화 횟수는 26 회이지만, 한 차례에 224 개 파일을 양산한 단계가 있어 토큰은 비선형적으로 분포.
- 가장 비용 효율적인 단계는 **2 차 양산** (파일당 약 1,100 토큰 — Python 생성기 + 템플릿 패턴 덕분).
- 가장 토큰을 많이 쓴 단계는 **4 차 시각화** (SVG 코드 양 + 57 개 슬라이드 일괄 주입 검증).
- **6 ~ 8 차는 `/compact` 약 2 ~ 3 회 발생. 컨텍스트 한도 초과 후 누적 추정**.

## 15. 시각 자료 의사결정 프레임 — 인포그래픽 vs 사진

4 차 시각화 작업과 5 차 이미지 도입 단계에서 **「언제 SVG 인포그래픽을 쓰고 언제 실사진을 쓰는가」** 의 기준이 자연스럽게 굳었습니다. 다른 과정에서도 이 프레임이 그대로 적용 가능합니다.

| 자리 | 어떤 시각 자료가 어울리는가 | 왜 |
|------|---------------------------|-----|
| **`.pain-point`** (이전 버전의 불편) | 실제 사진 (Unsplash CC0) | 「아픔」을 감정적으로 전달해야 함. 추상 도식은 오히려 거리감 |
| **`.new-tool`** (비유 도입) | 실제 사진 (카페·영수증·택시·칼통·자물쇠) | 비유의 출처가 일상 사물이므로 사진이 가장 직관적 |
| **개념 구조 설명** (Controller-Service-Mapper, MVC 6 계층, DI 효과) | SVG 인포그래픽 | 추상적 관계를 시각화. 사진으로 표현 불가 |
| **흐름도 슬라이드** (마지막 직전 고정) | SVG (`debug-flow`, `version-timeline`) | 데이터가 어디로 이동하는지 화살표·박스로 표현 |
| **Before / After 비교** | SVG 또는 좌우 분할 박스 | 「어디가 달라졌는가」 강조에 도식이 유리 |
| **확인 포인트(Checkpoint)** | 텍스트 + 작은 아이콘 | 학생이 따라하며 체크하는 자리 — 시각 부담 최소화 |

**핵심 원칙**: 비유 → 사진 / 구조 → 인포그래픽. **두 자료가 같은 슬라이드에 공존**해도 되며, `.new-tool` 의 사진 → 다음 슬라이드의 SVG 구조도 가 자연스러운 흐름.

`js/infographics.js` 의 12 개 SVG 는 「등록 패턴」 (`window.INFOGRAPHICS["client-server"] = (svg) => {...}`) + 공통 `PALETTE` 상수로 통일되어, 새 인포그래픽을 추가할 때 **색·서체가 자동으로 일관**됩니다.

## 16. 영구 에이전트 시스템

2026-05-01 후반 단계에서 `.claude/agents/` 에 **6 개 서브에이전트** 가 영구 등록되었습니다. 다음 세션부터 `subagent_type` 으로 직접 호출 가능하며, 한 세션 내에서 다회 호출되어 서로의 산출물을 검증·보강하는 협업 체인을 구성합니다.

| 에이전트 | 역할 | 주 호출 시점 |
|---------|------|-------------|
| `content-author` | 차시 본문 작성 (slides / examples / labs / handouts 4 종) | 새 차시 신설·보강 |
| `visual-designer` | SVG 인포그래픽·이미지 자리 배치 | 새 도식 필요 시 |
| `convention-auditor` | 학생 노출 금지 항목 (차시 번호·비전공자) / 비유 위치 / 4 단계 구조 점검 | 본문 작성 직후 |
| `flow-verifier` | 흐름도 슬라이드·미니맵·Checkpoint 3 종 누락 검사 | 본문 작성 직후 |

| `layout-verifier` | reveal.js 720 px 페이지 넘침 자동 검출 (`scripts/check-overflow.js`) | 슬라이드 작성 직후·전체 스캔 시 |

**사용 패턴**: 차시 한 개 신설 시 `content-author` → `visual-designer` → `convention-auditor` + `flow-verifier` + `layout-verifier` (병렬) → `documentation-curator` 순으로 연쇄 호출.

## 17. CODE-STANDARD 의 도입과 점진적 빌드업

콘텐츠 양산이 일정 규모에 달한 시점에서, **차시마다 패키지명·테이블명·필드명이 미세하게 달라지는 표류** 가 관찰되어 7 차에 `CODE-STANDARD.md` 를 도입했습니다.

**참고 출처**: `자료/` 의 `MavenMember`, `MavenBoard` 두 SMHRD JSP+Servlet 시대 프로젝트. 거기서 패턴을 추출해 현재 Spring + MySQL 스택에 맞춰 변환.

**핵심 결정**:

| 항목 | 표준 |
|------|------|
| 패키지 루트 | `com.smhrd.*` |
| 테이블명 | `mymember` / `myboard` / `myreply` |
| DB | MySQL (Oracle 시퀀스·ROWNUM 사용 안 함 → `AUTO_INCREMENT` + `LIMIT`) |
| 회원 도메인 필드 | `id, pwd, nick, email, joinedAt` |
| 게시판 도메인 필드 | `num, title, writer, content, createdAt, viewCnt` |

**「점진적 빌드업」 원칙** — 자료를 한 번에 완성형으로 보여주지 않습니다. 차시별로 `ALTER TABLE` / 리팩토링 단계를 거쳐 진화시킵니다.

| 버전 | 스키마/코드 진화 |
|------|---------------|
| v2 | `mymember(id, pwd)` — 평문 / 세션 없음 |
| v3 | `pwd` 컬럼 60 자 BCrypt 길이로 `ALTER` |
| v4 | 인터셉터 + 공통 레이아웃 (스키마 변화 없음) |
| v5 | `myboard(num, title, writer, content)` 신설 |
| v6 | `myboard` 에 `editedAt` 추가, 본인 확인 로직 |
| v7 | 페이징을 위한 인덱스 추가 |
| v8 | REST 컨트롤러 분리 (스키마 변화 없음) |
| v9 | `myreply(num, board_num, writer, content)` 신설 |
| v10 | `myboard` 에 `viewCnt` 컬럼·`myattach` 첨부 테이블 |

**적용 범위**: 4 Part 병렬 일괄 적용 — **108 개 파일** (auth 16 + board 12 + rest 28 + db/mvc 52).

## 18. 두 에이전트 충돌 사고와 교훈

team-collaboration 차시 보강 중에 **사용자 명확화 메시지를 별도 에이전트로 전달**한 사고가 있었습니다.

**경위**:

1. team-collaboration 본문 작업 중 사용자가 의도 명확화 메시지를 보냄 (회원가입은 예시일 뿐, 진짜 핵심은 5 단계 협업 흐름).
2. 그 메시지를 진행 중인 에이전트에 전달할 방법이 없어 **새 에이전트** 를 띄움.
3. 새 에이전트는 컨텍스트가 달라 메시지를 잘못 해석 → **엉뚱한 파일** (`auth-signup-form`) 을 ◆ 「프로젝트 개발 일면」 차시로 변경.
4. 원본 작업과 동시 진행되며 충돌이 발견됨.

**복구**:

- `auth-signup-form` 4 종 자료를 ▣ Part · AUTH 회원가입 폼 차시로 즉시 재작성.

**교훈**:

| 무엇이 문제였는가 | 다음번 대응 |
|------------------|-------------|
| 진행 중 에이전트에 SendMessage 불가 (현 도구 한계) | 새 에이전트는 가능한 한 띄우지 않음 |
| 동시 동작 두 에이전트가 같은 파일군을 만질 위험 | **한 번에 한 차시씩** 처리 |
| 의도 명확화가 늦으면 산출물이 다른 방향으로 굳음 | 작업 시작 전 의도 합의를 충분히 |

## 19. 이 프로젝트가 보여준 패턴

이 프로젝트는 **「AI 페어 프로그래밍 + 도메인 전문가 협업」** 의 한 사례입니다:

- **사용자(강사)** = 도메인 지식 + 학습자 이해 + 교수법 의사결정
- **Claude (Code)** = 빠른 양산 + 컨벤션 일관성 + 자동 검증
- **결과** = 5 시간 만에 완성된 56 차시 사이트

성공 요인:

1. **명확한 정렬 단계** — 본격 생산 전 충분한 협상
2. **반복 가능한 컨벤션** — 한 번 정한 패턴을 모든 차시에 적용
3. **즉각 피드백** — 잘못된 방향 발견 시 바로 수정 (예: 「비전공자」 노출, 비유 제목)
4. **점진적 깊이** — 첫 차시 충실 → 양산 → 핵심 차시 보강
5. **메모리 영구 저장** — 다음 세션에서도 컨벤션이 자동 적용

이 패턴은 다른 교육 콘텐츠 (Docker, Python, React 등) 에도 그대로 적용 가능합니다.
